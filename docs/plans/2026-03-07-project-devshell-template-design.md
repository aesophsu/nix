# Project DevShell Template Design

**Date:** 2026-03-07

## Goal

Build a machine-level convention for project-local Nix development environments:

- keep a small global "base tooling" layer in Home Manager
- make each project own its `flake.nix` + `.envrc`
- enable `direnv` + `nix-direnv` so entering a project directory activates the project shell
- provide reusable templates and helper commands for both new projects and freshly cloned repositories

## Current State

- Global packages are split between:
  - `user/common/core/packages.nix` for base CLI and `direnv`
  - `user/common/core/tooling/*.nix` for language/toolchain packages
- `programs.direnv.enable = true` and `programs.direnv.nix-direnv.enable = true` are already enabled.
- Repository root already uses `.envrc` with `use flake`.
- Shell profiles do not yet explicitly show a `direnv hook` integration point.

## Design

### 1. Two-layer environment model

Use a strict two-layer model:

- Global base tooling layer:
  - only stable, cross-project CLI and shell integration
  - includes `direnv`, `nix-direnv`, Nix inspection tools, and helper scripts
- Project-local environment layer:
  - every project declares its own `flake.nix`
  - `.envrc` stays minimal and only delegates to `use flake`
  - language/toolchain dependencies move toward project-local declaration

This avoids hidden global dependencies and keeps project environments reproducible.

### 2. Template-based project bootstrap

Add a template directory in this repo, for example `templates/devshell/`, with:

- `base`: minimal `flake.nix` + `.envrc`
- `python`: `base` plus Python/uv/ruff shell packages
- `node`: `base` plus Node/Corepack shell packages

Templates should be small, explicit, and easy to copy into an existing repository.

### 3. Helper commands

Add a lightweight helper script exposed globally, with two main workflows:

- initialize a new or empty project directory from a named template
- add missing `.envrc` and/or `flake.nix` to an existing cloned repository

Proposed commands:

- `devshell-init <template>`
- `devshell-attach <template>`

Behavior constraints:

- never overwrite existing `flake.nix` or `.envrc` by default
- fail clearly when target files already exist
- print the exact next step: `direnv allow`

### 4. Automatic activation flow

Activation path:

1. shell starts with `direnv hook` loaded
2. user `cd`s into a directory
3. `direnv` reads `.envrc`
4. `use flake` resolves the project `devShell`
5. environment activates after `direnv allow`

This supports:

- hand-created projects
- cloned GitHub repositories that already include Nix files
- cloned repositories retrofitted with local templates

### 5. Convergence on layer boundaries

Two deliberate follow-up adjustments are included:

- weaken the role of globally installed language toolchains
- prefer language/toolchain packages in project templates instead

This does not require deleting useful global tooling immediately. The implementation should make the template path the default and document the boundary clearly.

## Testing Strategy

Verify:

- Home Manager evaluation still succeeds after shell integration changes
- helper scripts are installed and runnable from the shell
- each template evaluates with `nix eval` or `nix develop`
- `.envrc` generated by templates is accepted by `direnv allow`
- helper scripts do not overwrite pre-existing project files

## Risks And Mitigations

- Risk: duplicated responsibility between global and project toolchains
  - Mitigation: document the boundary and trim global packages where safe
- Risk: helper scripts accidentally overwrite user files
  - Mitigation: default to create-only behavior and explicit failure
- Risk: shell hook changes affect existing shell startup
  - Mitigation: use Home Manager's standard `direnv` integration path and verify with evaluation

## Success Criteria

- entering a correctly prepared project directory activates the project shell automatically
- new projects can be bootstrapped from a template in one command
- existing cloned repositories can be attached to the convention with one command
- project-local environments become the default place for language/runtime declarations
